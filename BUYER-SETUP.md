# FREQUENCY — Buyer setup guide

You just bought the full source code for FREQUENCY. This guide gets you from
"zip file" to "live app you can test" in about 15 minutes, using only **free
tiers** — no credit card required on any of the three services.

The stack:

| Piece | Service | Folder in this repo | Cost |
|---|---|---|---|
| Database (Postgres) | [Neon](https://neon.tech) | — (schema auto-migrates) | Free |
| Backend API (Express) | [Render](https://render.com) | `frequency/server/` | Free |
| Frontend (Vite + React PWA) | [Vercel](https://vercel.com) | `frequency/` | Free |
| Google sign-in *(optional)* | [Google Cloud](https://console.cloud.google.com) | — | Free |

Deployment config is already in the repo — you don't write any of it:

- `render.yaml` — Render Blueprint (backend). Auto-generates `SESSION_SECRET` for you.
- `frequency/vercel.json` — Vercel build config (frontend).
- `frequency/server/migrations/` — database schema, applied automatically on every backend start.

> **Full step-by-step walkthrough with screenshots-level detail:** see
> [`SETUP.md`](SETUP.md). This file is the condensed checklist + the complete
> environment-variable reference.

## Try it locally first (no accounts needed)

The app degrades gracefully — it runs with **zero** configuration:

```sh
# terminal 1 — backend
cd frequency/server && npm install && npm run dev   # http://localhost:8787

# terminal 2 — frontend
cd frequency && npm install && npm run dev          # http://localhost:5173
```

Open http://localhost:5173 and play. Without a database it uses built-in
fallback content; accounts/persistence light up once you add `DATABASE_URL`.

Requires Node.js **20.12+**.

## Deploy checklist (in this order)

1. **Neon** — create a project, copy the connection string
   (`postgresql://...?sslmode=require`). That's your `DATABASE_URL`.
2. **Render** — New + → **Blueprint** → pick this repo. Render reads
   `render.yaml` and creates the `frequency-api` web service.
   - `SESSION_SECRET` is generated automatically by the Blueprint.
   - Set `DATABASE_URL` to the Neon string.
   - Leave `FRONTEND_ORIGIN` blank for now (step 4).
   - Note your Render URL, e.g. `https://frequency-api.onrender.com`.
   - Verify: `https://<render-url>/api/health` → `{"ok":true,"persisted":true}`.
3. **Vercel** — Add New → Project → import this repo.
   - **Set Root Directory to `frequency`** (the one setting that matters).
   - Add env var `VITE_API_URL` = your Render URL.
   - Deploy. Note your Vercel URL, e.g. `https://frequency.vercel.app`.
4. **Connect them** — back on Render, set `FRONTEND_ORIGIN` = your Vercel URL
   (exact, no trailing slash). Render redeploys automatically. Done.
5. **Google sign-in** *(optional, skippable)* — create an OAuth Web client in
   Google Cloud, then set the same client id as `GOOGLE_CLIENT_ID` on Render
   and `VITE_GOOGLE_CLIENT_ID` on Vercel, and redeploy the frontend.
   The "Continue with Google" button stays hidden until both are set.

## Complete environment variable reference

These are **all** the environment variables in the entire codebase — there are
no hidden ones.

### Backend — Render (or `frequency/server/.env` locally)

| Variable | Required? | What it is / where to get it | Example |
|---|---|---|---|
| `DATABASE_URL` | Yes, for accounts & persistence (app still runs without it in fallback mode) | Neon connection string, must include `?sslmode=require` | `postgresql://user:pass@ep-xxx.aws.neon.tech/neondb?sslmode=require` |
| `SESSION_SECRET` | Yes, for logins | Auto-generated by the Render Blueprint. Locally: `node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"` | 64-char random hex |
| `FRONTEND_ORIGIN` | Yes, in production (CORS + cross-site login cookies) | Your Vercel URL, no trailing slash. Comma-separate multiple domains | `https://frequency.vercel.app` |
| `GOOGLE_CLIENT_ID` | Only for Google sign-in | Google Cloud Console → Credentials → OAuth client ID (Web application) | `1234-abc.apps.googleusercontent.com` |
| `PORT` | No | Render sets this automatically; defaults to `8787` locally | `8787` |

Template: [`frequency/server/.env.example`](frequency/server/.env.example)

### Frontend — Vercel (or `frequency/.env` locally)

| Variable | Required? | What it is / where to get it | Example |
|---|---|---|---|
| `VITE_API_URL` | Yes, in production (locally the dev server proxies `/api` automatically) | Your Render backend URL | `https://frequency-api.onrender.com` |
| `VITE_GOOGLE_CLIENT_ID` | Only for Google sign-in | The **same** client id as `GOOGLE_CLIENT_ID` above | `1234-abc.apps.googleusercontent.com` |

Template: [`frequency/.env.example`](frequency/.env.example)

> **Vite variables are baked in at build time.** If you add or change a
> `VITE_...` value on Vercel, you must redeploy the frontend for it to take
> effect.

### Minimum set to test the deployed app

For a buyer who just wants to verify everything works end-to-end:

```
Render:  DATABASE_URL, SESSION_SECRET (auto), FRONTEND_ORIGIN
Vercel:  VITE_API_URL
```

Google sign-in is entirely optional — callsign + password accounts work
without it.

## Quick verification after deploy

1. `https://<render-url>/api/health` returns `{"ok":true,"persisted":true}`.
2. Open your Vercel URL — the game loads and plays.
3. On the intro screen footer, click "claim a callsign", create an account,
   reload — you stay signed in. (If login doesn't stick, `FRONTEND_ORIGIN` is
   wrong — see the [SETUP.md troubleshooting table](SETUP.md#troubleshooting).)

**Free-tier note:** Render free services sleep after ~15 minutes idle and take
~30–60 s to wake. The app stays playable (fallback mode) while it wakes — this
is normal and disappears on any paid Render plan.
